Vision

TEAMMATES is a tool to help teachers manage student teams.

Instead of providing ‘everything’ needed for managing teams, we provide features specific to student teams which are typically absent in industry team management tools.

Our target users are teachers running team-based courses. Their students are secondary users. i.e., Teachers decide to use the system, not students.

Supplementary requirements

Point Calculation Scheme

  1. Students enter values using the contribution scale (see glossary).

Based on the values above, we try to deduce the student’s answer to the following two questions:

(a) In your opinion, what portion of the project did you do?

(b) In your opinion, if your teammates are doing the project by themselves without you, how do they compare with each other in terms of contribution?

  1. Output values are calculated as follows:

0.  normalize claimed values .
(normalized value) = (original value) * (bump ratio)

e.g.,

entered values  90 [self], 110,130, N/A (total = 330)

bump ratio: (100*3)/(90+110+130) = 300/330

normalized: 82, 100, 118, N/A

normalized total = 300 (100*number of ratings)

This answers the question (a) above. The student thinks he did ‘Equal share - 18%’ (as indicated by 82).

1. Calculate unbiased values by removing self-rating bias

Ignore the self rating and normalize remaining values.

e.g.,

normalized input (from above): 82,100,  118, N/A

Calculating unbiased values:

82 → ignored.

100 →  100*200/(100+118) = 92

118 →  118*200/(100+118) = 108

Unbiased values: [self], 92, 108, N/A

Unbiased values total = 200 (100*number of ratings)

This answers the question (b) above. In the example above, the student thinks his teammates contribution ratio is 92:108 and is unsure of the third teammate.

2. calculate averagePerceived: For each student, take the average of unbiased values that others have given him.

3. calculate normalizedAveragePerceived: normalize the averages, similar to how input was normalized.

normalizedAveragePerceived = averagePerceived*bumpRatio

bumpRatio = 100 * (number of students with averagePerceived values)/(sum of averagePerceived)

This is the relative work distribution among team members based on unbiased opinions of team members.

4. For each student, denormalize normalizedAveragePerceived

We scale back to match the total of claimed by student. That way, student can compare his input (i.e., his opinion of the team’s work distribution) with what the team thinks of the same.
In the example used above, we should use 330/300 as the denormalizing factor for that student. The result could be something like this

student’s opinion: 90 [self], 110,130, N/A (total = 330)

team’s opinion   :  95,          105,125, 115 (total = 440)

Value transformation steps: claimed → normalizedClaimed → unbiasedClaimed → averagePerceived → normalizedAveragePerceived → denormalizedAveragePerceived

  1. Student view:
  1. for claimed contribution, show: same as what the student entered initially (otherwise, the student will be confused as to how the value got changed)
  2. for perceived contribution, show: denormalizedAveragePerceived 
  1. Instructor view:
  1. for claimed contribution, show: normalizedClaimed 
  2. for perceived contribution, show:  normalizedAveragePerceived 

Note: 

Scenario 1: If students give 0 points to each other, then everyone should receive Equal Share and difference should be 0.

Scenario 2: If students are not sure or do not submit the evaluation, then Perceived/Claimed for Instructor should be shown as N/A instead of Equal Share. In this case, difference should be shown as N/A also.

Report format

Some important questions a instructor may want answered (i.e., the system should provide ways of answering these quickly and easily)

- This guy's average is rather high. Did everyone give him positive points or was the average skewed by his friend? {This is addressed by SummaryReport}

- This guy got very bad comments. What did he say about others (may be there were mutual bad feelings between him and the team)? {This is addressed by DetailedReport ByStudent}

- Who overestimated their contribution? {Can be done by sorting based on [claimed-perceived]}

- Who are high/low contributors? {Can be done by sorting based on perceived}

Reports have to be highly optimized for space, navigation, and attention.

  1. space: should not have very long reports with lot of white space. This is particularly a problem if we use table format and one cell happens to contain a large blob of data like a super long rant by a student.
  2. navigation: instructors should be able to find information easily without having to scroll up and down. think of locality of information. data that are related should appear together.
  3. attention: the report should highlight important things and provide ways to zoom in important things. for example, instructor should be able to find the cases that require his attention most. using colors, sorting facilities, etc can help here.

Scalability

In phase 1, the system should be able to handle

  1. Classes of up to 300 students
  2. Up to 10,000 students
  3. Up to 100 instructors

Security

  1. Keys are not too similar to each other {to avoid ‘guessing’ of keys}
  2. Prevents masquerading via URL manipulation
  3. Immune to DoS attacks.
  4. Immune to cross-site and SQL injection attacks
  5. Only the person who deployed the system has the key to access the API directly.

  1. DetectBouncing: detect emails that bounced and let instructor know
  2. ShowInviteSent:  Show if the student was sent an invite.
  3. DirectUrlToPage: Most pages in the system can be reached using a direct URL  
  4. TrackReportViews: Keep track of whether students actually view published reports. It may be useful to identify those who don’t even care enough to read feedback from peers.
  5. SelectiveRelease: Instructor can block release of reports for particular student (e.g., because that did not write good comments).
  6. PeerFeedbackOnly: Evaluations can be set up in a way that students can feedback each other without revealing their feedback to instructors
  7. ConfigurableMessages: Messages shown to users (including email messages) are editable by the administrator, instructors
  8. Installer: An installer to install Teammates on a given GAE account {This is for those who do not want their data on our GAE account}
  9. InstructorSelfRegister: Instructors can register in the system without having to contact Teammates instructor for a new account
  1. RunningEvaluationCount: Home page shows the total evaluations done to date (e.g., 3457 evaluations so far and counting...) e.g.  http://skitch.com/
  1. MultiEvaluationReports: Reports that combines information from multiple evaluations in the same course
  2. RespondToFeedback: Students can respond to anonymous feedback by teammates
  3. AdviceByInstructor: Instructor can add his own advise before publishing the report
  4. DiscussionForum: A place for users to discuss issues (e.g., a GoogleGroup)
  5. SelfGuidedWorkflow: Users can use the system with minimal documentation. The system guides the user to the next action.
  6. FindReportsForStudent: To be able to find evaluation reports of a particular student
  7. Internationalization
